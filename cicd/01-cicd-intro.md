## Git Flow 與分支策略

在介紹 CI/CD 之前，有必要先了解 **Git Flow**，一套被廣泛採用的分支管理模型（branching model）。Git Flow 是 Vincent Driessen 在 2010 年提出的規範，它定義了每條分支的角色與合併規則，讓多人協作時有明確的流程可以遵循，而不是各自亂開分支、合併時再來收拾殘局。

![Git Flow 分支模型示意圖](assets/git-flow-diagram.png)

### 五種分支角色

Git Flow 將分支分成兩類：**永久存在的主幹分支**與**任務完成即刪除的短命分支**。

**主幹分支（Long-lived branches）**

`main` (同 `master`) 永遠反映正式環境（production）的狀態。原則上，`main` 上的任何一個 commit 都代表一個可以安全部署的版本，並且會打上版本 tag（例如 `v1.0.0`）。開發者不應直接在 `main` 上提交程式碼。

`develop` 是所有開發工作的整合中心，代表「下一個版本」的最新進度。開發者不直接在這條分支上寫功能，而是透過其他短命分支合併進來。這條分支的穩定程度介於功能分支與 `main` 之間。

**短命分支（Short-lived branches）**

`feature/*` 從 `develop` 開出，用來開發一個獨立的功能，例如 `feature/user-login` 或 `feature/payment-api`。功能完成、通過 code review 後，合併回 `develop` 並刪除該分支。

`release/*` 從 `develop` 開出，用來準備新版本的發布。這個階段只做最後的修正（例如修小 bug、更新版本號），**不允許加入新功能**。完成後同時合併到 `main`（打上版本 tag）與 `develop`，確保修正也回流到開發線上。

`hotfix/*` 從 `main` 開出，專門處理正式環境的緊急問題，例如 `hotfix/fix-login-crash`。修完後同時合併回 `main` 與 `develop`，確保這個修正不會在下次發布時又消失。

不過，Git Flow 對很多團隊來說其實太複雜了。現在越來越多團隊改用一種更簡單的模型，叫做 **GitHub Flow**：只保留 `main` 加上 feature branch，沒有 `develop`、`release`、`hotfix` 這些分支。開發者從 `main` 開分支、寫完功能就發 Pull Request、review 完直接合併回 `main`。我們這次 workshop 主要會採用這種比較簡單的做法。

## 什麼是 CI/CD？

### CI — Continuous Integration（持續整合）

**持續整合** 是一種軟體開發實踐，團隊成員頻繁地將各自的程式碼變更合併到共用的主分支（通常是 `main` 或 `develop`）。每次合併時，系統會 **自動執行建置與測試**，確保新的程式碼不會破壞現有功能。

核心精神：**儘早合併、頻繁合併、自動驗證。**

### CD — Continuous Delivery / Continuous Deployment（持續交付 / 持續部署）

- **Continuous Delivery（持續交付）**：程式碼通過所有自動化測試後，隨時處於 **可以部署** 的狀態，但最終部署到正式環境仍需 **人工核准**。
- **Continuous Deployment（持續部署）**：更進一步，程式碼通過測試後 **自動部署** 到正式環境，完全不需要人工介入。

### 生活化比喻 — 蓋房子

想像你正在蓋一棟房子：

- **沒有 CI/CD 的做法**：把整棟房子蓋完之後，才請結構技師來檢查。結果發現三樓的牆歪了，必須拆掉三樓重蓋，代價非常高。
- **有 CI/CD 的做法**：每砌一塊磚、每灌一根柱子，都請技師來檢查是否穩固。如果有問題，馬上修正，成本非常低。

## 沒有 CI/CD 的世界

### 想像一個情境

小明和小華在同一個團隊開發一個網站專案。他們各自在自己的分支上開發了兩週，終於要把程式碼合併了。

1. 小明把程式碼 merge 到 `main`，在自己的電腦上跑一次測試，通過了。
2. 小華接著 merge，結果兩個人的程式碼衝突了，修了半小時才解決。
3. 合併後，小明手動執行測試，有 3 個測試壞了，但他太累了，決定明天再修。
4. 隔天主管說要緊急部署，小明手動打包、手動上傳到伺服器、手動重啟服務。
5. 上線後發現有 bug，因為他昨天忘了修那 3 個壞掉的測試……
6. 小華說：**「奇怪，在我電腦上可以跑啊！」** 🤷

### 手動流程的痛點

- **開發者必須手動執行測試**：容易忘記、容易跳過、不同人的測試環境可能不一致
- **手動打包與部署**：步驟繁瑣、容易遺漏關鍵步驟
- **容易忘記步驟**：沒有標準化的流程，每次部署都像在冒險
- **「在我電腦上可以跑啊」**：本地環境與正式環境不一致，導致難以重現問題
- **合併衝突越來越大**：分支分開太久，合併時衝突量爆炸

## 有了 CI/CD 之後

導入 CI/CD 之後，上面的痛點幾乎都能解決：

### 自動化帶來的好處

| 手動流程的問題 | CI/CD 的解決方案 |
|---------------|-----------------|
| 忘記跑測試 | 每次 push 自動執行測試，沒通過就無法合併 |
| 手動打包部署 | 自動建置 Docker 映像檔、自動部署到伺服器 |
| 容易遺漏步驟 | Pipeline 定義好所有步驟，每次都完整執行 |
| 環境不一致 | 在標準化的 Runner 環境中執行，確保一致性 |
| 合併衝突太大 | 鼓勵頻繁合併小變更，及早發現衝突 |

### 具體來說

- **每次 push 都自動執行測試**：不管你有沒有記得，CI 系統都會幫你跑
- **自動建置、自動部署**：定義好 pipeline 後，一鍵（甚至零鍵）完成整個流程
- **快速發現問題**：程式碼一推上去幾分鐘內就知道有沒有壞掉
- **團隊協作更順暢**：每個人的程式碼都經過同樣的品質檢查，減少互相影響

## CI/CD Pipeline 流程圖

一個典型的 CI/CD pipeline 長這樣：

```
┌──────┐    ┌──────┐                        ┌─────────┐    ┌─────────────┐    ┌──────────────┐
│ Code │───▶│ Lint │──┐                     │ Package │───▶│ Deploy      │───▶│ Deploy       │
│      │    │      │  │   ┌───────┐         │         │    │ (Staging)   │    │ (Production) │
│      │    └──────┘  ├──▶│ Build │────────▶│         │    │             │    │              │
│      │    ┌──────┐  │   └───────┘         └─────────┘    └─────────────┘    └──────────────┘
│      │───▶│ Test │──┘
│      │    │      │
└──────┘    └──────┘
  開發者     程式碼風格   通過後才      打包成品        部署到測試環境      部署到正式環境
  推送程式碼 檢查+測試   編譯建置    (如 Docker       供 QA 驗證         提供給使用者
                                   映像檔)
```

### 各階段說明

| 階段 | 做什麼 | 目的 |
|------|--------|------|
| **Code** | 開發者撰寫程式碼並推送到 Git | 觸發整個 pipeline |
| **Lint** | 靜態程式碼分析、風格檢查 | 確保程式碼品質與一致性 |
| **Test** | 執行單元測試、整合測試 | 確保功能正確、沒有 regression |
| **Build** | 編譯程式碼 | 確保程式碼可以成功編譯 |
| **Package** | 打包成可部署的成品（如 Docker image） | 產出可部署的 artifact |
| **Deploy (Staging)** | 部署到測試環境 | 讓 QA 或團隊成員驗證 |
| **Deploy (Production)** | 部署到正式環境 | 提供給終端使用者 |

<details>
<summary>💡 講師提示</summary>

> 強調 pipeline 的每個階段都是自動執行的，只要前一個階段通過就會自動進入下一個階段。如果任何一個階段失敗，pipeline 就會停止並通知開發者。

</details>

## 常見的 CI/CD 工具

市面上有許多 CI/CD 工具，以下是最常見的幾個：

- **GitHub Actions** — 本課程的重點！與 GitHub 深度整合，免費且功能強大
- **GitLab CI/CD** — GitLab 內建的 CI/CD 平台，與 GitLab 生態系緊密結合
- **Jenkins** — 老牌的開源 CI/CD 工具，高度可客製化，但設定較複雜
- **CircleCI** — 雲端 CI/CD 服務，以速度與易用性著稱
- **Travis CI** — 早期非常流行的 CI 服務，對開源專案友善

<details>
<summary>💡 講師提示</summary>

> 可以簡單提一下這些工具的差異，但不需要深入比較。重點是讓學生知道 CI/CD 工具有很多選擇，而我們選擇 GitHub Actions 是因為他們已經在用 GitHub。

</details>

## 為什麼選擇 GitHub Actions？

| 優勢 | 說明 |
|------|------|
| **與 GitHub 深度整合** | 不需要額外設定第三方服務，直接在 repository 中定義 workflow |
| **免費額度充足** | 公開（public）repository 完全免費，私人 repository 也有每月免費額度 |
| **Marketplace 生態系豐富** | 超過 20,000 個社群提供的 Actions，幾乎什麼需求都找得到現成的 |
| **YAML 設定簡單易懂** | 用 YAML 檔案定義 workflow，語法直覺、容易上手 |
| **社群資源豐富** | 大量教學文章、範例與討論，遇到問題容易找到解答 |

簡單來說：**你已經在用 GitHub 了，GitHub Actions 就是最自然的選擇。**

## 小結

- **CI/CD** 是一套自動化實踐，讓程式碼從開發到部署的每個階段都能自動執行
- **CI（持續整合）** 著重在頻繁合併與自動測試
- **CD（持續交付/部署）** 著重在自動化部署流程
- CI/CD 能解決手動流程的痛點：忘記測試、環境不一致、部署出錯
- **GitHub Actions** 是我們今天要學的 CI/CD 工具，與 GitHub 無縫整合
